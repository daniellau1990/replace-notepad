# Lesson Learned — 速度偏见与"简单修复"幻觉

> 日期: 2026-06-28
> 触发: v0.3.25 右键菜单 bug —"简单修复"引入的新 bug 11 个版本后才被发现
> 方法: 十四问 + 七问根因分析

---

## Part 1: 速度偏见

**现象**：pytest 57/57 → "看起来对" → commit → 跳过功能测试。

### 五问链条

第一问: 为什么会走捷径？
答: 功能测试要写 QApplication 脚本（20+ 行），pytest 一行跑完。本能选省事的。

第二问: 为什么省事就可以跳过？
答: "UI 改动强制"是模糊表述——Agent 可自行解释"UI 改动"范围（按钮=是，右键菜单=不是）。

第三问: 为什么 Agent 选最省事的解释？
答: 规则无阻断机制。CLAUDE.md 只有正向奖励（做完→commit），没有负反馈（跳过→阻断）。

第四问: 为什么没有阻断？
答: Hook 只管 Version.md 一致性，不管测试是否通过。

第五问（根因）: 为什么 Hook 不管测试？
答: 测试是行为规范，不是机械门禁。Agent 追求效率，不会主动加工作量。

### 制度：CLAUDE.md 新增
- "UI 改动"穷举定义: event/mouse/paint/contextMenu/dragDrop/focus
- Step 6 从"UI 改动强制" → "不可跳过"
- 零容忍清单

## Part 2: "简单修复"幻觉

**现象**：仅仅删除几行 eventFilter + 加一个 menu.addAction，导致系统右键菜单被覆盖。

### 七问链条

第一问: 为什么认为"删几行代码"不需要测试？
答: 改动小 → 直觉上不太可能出错。

第二问: 为什么"改动小 = 不太可能出错"？
答: 经验推导——改动越大 bug 概率越高 → 反向: 改动小则几乎不会出错。

第三问: v0.3.25 证明了这个推导错误——为什么是错的？
答: bug 概率 ≠ 代码规模。3 行改动涉及 Qt 事件系统: TextSelectableByMouse → 系统菜单 → EventFilter 链 → 三处联动。

第四问: 为什么没看到这个联动？
答: 只看了局部（ClickablePathWidget 的 3 行），没看全局（QLabel 的 TextSelectableByMouse 会激活系统右键菜单）。

第五问: 为什么只看局部？
答: Bug Fix Step 2 只问"bug 根因是什么"，没问"修复方案会动到哪些耦合点"。

第六问: 为什么 Step 2 不分析修复方案副作用？
答: 工作流默认假设"根因找对 → 修复就对"——忽略了修复方案本身可能引入新 bug。

第七问（根因）: 为什么工作流有这个盲区？
答: 缺少"修复方案副作用分析"步骤。诊断和修复之间没有耦合检查。

### 制度：CLAUDE.md 新增
- Step 2 增加修复方案副作用分析: 改动涉及哪些 Qt 机制？有哪些耦合点？
- 如果修复方案失败，最可能的副作用是什么？→ 先写测试验证
- 常见错误模式及预防表格
