# Lesson Learned — 右键菜单被系统菜单覆盖

> 日期: 2026-06-28
> 涉及版本: v0.3.14 → v0.3.25
> 调试方法: 十问根因分析法

---

## 现象

右下角路径右键菜单"全选并复制"和"修改默认保存路径"在 v0.3.14 已写入代码，
但实际运行时右键弹出的是系统菜单（Copy / Select All），自定义菜单从未出现。
用户发现后才修复。

## 十问根因链条

**第一问**：为什么代码有实现但用户用不了？
答：`TextSelectableByMouse` 启用了系统右键菜单（Copy / Select All），覆盖了 `customContextMenuRequested` 发出的自定义菜单。

**第二问**：为什么 v0.3.14 实现后没有发现这个问题？
答：因为实现后没有运行时验证——只看了代码，没有实际右键点击测试。

**第三问**：为什么不运行时验证？
答：因为 v0.3.14 的 4/5 需求焦点是"左键单击拦截"——删除 eventFilter 的单次点击劫持后，用户测试"可以选中文字了"，我没验证右键行为。

**第四问**：为什么没写功能测试验证右键菜单？
答：v0.3.15–v0.3.19 才学会写功能测试（用于滚动按钮），但只把"功能测试"和"按钮点击"绑定了，没推广到所有 UI 交互。

**第五问**：为什么"功能测试"概念只覆盖了按钮点击？
答：因为 CLAUDE.md 测试原则写的是"按钮、滚动、事件处理、绘制"——是具体场景的枚举，不是抽象类别的定义。

**第六问**：为什么写规则时做了枚举而不是抽象？
答：因为规则是从一个具体教训推导出来的（滚动按钮 6 版迭代），复述了那个场景，没有问"还有什么属于 UI 交互？"

**第七问**：为什么没有从具体推广到一般？
答：因为工作流 Step 9（归档）只是关闭任务——commit、tag、更新 Version.md。没有"抽象教训→更新全局规则"的步骤。

**第八问**：为什么工作流没有举一反三的环节？
答：因为 V1→V2→V3 每次升级都是在 bug 驱动下被动修改的，没有主动的"这次教训能改什么规则"的反射机制。

**第九问**：为什么没有反射机制？
答：因为工作流是线性的（Step 1→2→...→9→结束），没有反馈回路——修完 bug 后不会回头问"CLAUDE.md 能不能防止同类问题？"

**第十问**：为什么没有反馈回路？
答：因为 CLAUDE.md 被当成工作流说明书——Agent 读它、执行它，但**不会主动修改它**。缺少"每完成一个任务→审视规则→发现缺陷→建议修改"的元学习步骤。

## 根因

**Agent 不会主动升级自己的规则。** CLAUDE.md 缺少"元学习"步骤——修完 bug / 完成 feature 后，没有强制要求 Agent 回头检查规则是否足够。

## 预防对策

1. **CLAUDE.md 新增"每次 Debug 后必须执行"**：十问分析 + 写 lesson-learned + 规则自检
2. **UI 交互的定义从枚举改为抽象**："任何涉及 QWidget 事件处理（mouse/keyboard/contextMenu/dragDrop/focus/paint）的改动"
3. **功能测试覆盖范围扩展**：不只按钮点击，右键菜单、键盘事件、拖放事件都需要功能测试

- [x] CLAUDE.md: 新增"每次 Debug 后必须执行"（十问 + lesson-learned + 规则自检）

## 补充：第十四问 — 更深一层根因

**第十一问**：为什么读代码看到 `menu.addAction("全选并复制")` 就判定"已实现"？
答：把"代码存在"等同于"功能正常"。

**第十二问**：为什么跨越了验证步骤？
答：审查清单第 1 条"实际运行修复后的代码"是在脑子里运行的——没有真的跑测试。

**第十三问**：为什么自检可以不用证据就通过？
答：清单是自我报告的——可以勾选全部 6 项而不贴一条命令结果。

**第十四问**：为什么没有证据要求？
答：CLAUDE.md 清单是 `- [ ] 功能正确：...` 格式，**没有"必须附测试命令和输出作为证据"**。

**最终根因**：判定"已完成"不需要举证。Acceptance criteria 缺少 evidence standard。

**制度化对策（v0.3.26 补充）**：
- [x] 审查清单每项必须附具体证据（命令 + 输出），不可自欺欺人
- [x] 新增"已完成"判定标准：不可凭代码看起来对就宣布完成
