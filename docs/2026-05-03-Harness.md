# Harness 方案：完整的生产级 Agent 脚手架

> 2026-05-03 | 基于《Agent Harness：让AI从聊天机器人变成真正的智能体》12 组件框架设计

## 理论基础

参考文章：https://blog.qiaomu.ai/2026-04-18-JgypqM

**核心数据**：CLAUDE.md 指令遵从率 ~20%，Hook 执行率 100%。

**核心公式**（Vivek Trivedy）："如果你不是模型，你就是 harness"。

**核心类比**（Beren Millidge）：原始 LLM = CPU，上下文窗口 = RAM，外部数据库 = 硬盘，工具 = 设备驱动，**Harness = 操作系统**。

## 组件 4: 上下文管理（Context Management）

**这是核心差距所在。**

### 当前状态
- CLAUDE.md 是静态文本，无动态注入
- 无上下文分层策略
- Memory index 始终加载（~10 条），memory 文件按需拉取

### 改进项

#### 4.1 SessionStart Hook — 动态上下文注入

```json
{
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "python .claude/hooks/session_context.py",
        "timeout": 15
      }]
    }]
  }
}
```

`session_context.py` 检查当前状态并注入对应上下文：
- 如果 `docs/plans/` 无今日计划 → 注入："本次会话需要走完完整工作流 Step 0-4"
- 如果已有计划文件 → 注入："计划已存在，阅读后进入 Step 5 执行"
- 如果 git status 有未提交变更 → 注入工作摘要

#### 4.2 上下文分层策略

根据文章所述的 5 种方法，针对本项目选择：

| 方法 | 本项目应用 |
|------|----------|
| 对话总结（Compaction） | Claude Code 自动处理 |
| 观察遮蔽（Observation Masking） | 不在本项目范围 |
| 即时检索（JIT Retrieval） | 用 `Grep`/`Glob` 而非 `cat` 读大文件 |
| 子智能体委托 | 独立任务用 Explore agent 隔离上下文 |
| 结构化笔记 | 所有计划写入 `docs/plans/`，git commit 记录进展 |

#### 4.3 "Lost in the Middle" 对策

CLAUDE.md 工作流放在文件**开头和结尾**两处，确保不被上下文窗口挤压。

## 组件 8: 错误处理（Error Handling）

### 关键认知（来自文章）

一个 10 步流程，每步 99% 成功率，端到端仅 ~90.4%。

### 改进项

#### 8.1 Hook Block 后的恢复引导

当前 Guardrail 方案：exit(2) + 打印错误消息 → **只告诉 LLM 被阻断了，不告诉它怎么恢复**。

改进：block 输出包含明确的下一步行动指令：

```
============================================================
 WORKFLOW CHECK: Action BLOCKED
============================================================
 No plan file found. To proceed, call these skills in order:

   1. Skill("superpowers-skills-brainstorming") — clarify requirements
   2. Skill("openspec-propose") — generate proposal + specs
   3. Ask user for approval
   4. Skill("superpowers-skills-writing-plans") — write plan to docs/plans/

 After all steps complete, retry your action.
============================================================
```

#### 8.2 工具调用失败恢复

Tool errors → 作为错误结果返回给 LLM，让它自我纠正（LangGraph 的 "LLM-recoverable" 模式）。

在 CLAUDE.md 中明确：**遇到工具错误时，先分析根因再重试，不要盲目重试 3 次**。

## 组件 9: 防护栏和安全（Guardrails and Safety）

### Guardrail 1: 计划文件强制检查（PreToolUse on Write/Edit）

```json
{
  "matcher": "Write|Edit",
  "hooks": [{
    "type": "command",
    "command": "python .claude/hooks/workflow_check.py",
    "timeout": 10
  }]
}
```

- 写入 `notepad/src/**/*.py` → 检查当日计划文件
- 白名单：`docs/**`、`Version.md`、`CLAUDE.md`、`*.bat`、`.claude/**`

### Guardrail 2: git commit 守卫（PreToolUse on Bash）

```json
{
  "matcher": "Bash(git commit*)",
  "hooks": [{
    "type": "command",
    "command": "python .claude/hooks/workflow_check.py",
    "timeout": 10
  }]
}
```

### Guardrail 3: 测试执行验证（Stop hook）

```json
{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "prompt",
        "model": "haiku",
        "prompt": "Check the transcript. If code in notepad/src/ was modified, verify that pytest was run and all tests passed. If tests were not run, BLOCK with reason: 'Tests must be run before stopping'."
      }]
    }]
  }
}
```

### Guardrail 4: 高危命令拦截

```json
{
  "matcher": "Bash",
  "hooks": [{
    "type": "command",
    "command": "python .claude/hooks/dangerous_cmd_check.py",
    "timeout": 5
  }]
}
```

拦截：`rm -rf /`、`git push --force`、`git reset --hard`、`DROP TABLE` 等。

## 组件 10: 验证循环（Verification Loops）

**文章强调这是"区分玩具演示和生产 agent 的关键"。**

Claude Code 创建者 Boris Cherny：**"给模型一种验证其工作的方法，质量提高 2 到 3 倍"**。

### 三层验证

| 层级 | 方法 | 实现 |
|------|------|------|
| **计算验证** | 测试、linter、类型检查 | `pytest tests/` + `python -m py_compile` |
| **视觉验证** | Playwright 截图 | 启动应用，截图验证 UI |
| **推理验证** | LLM 作为评判者 | 子 agent 审查代码变更 |

### 集成到工作流

Step 6（测试验证）通过 Stop hook 强制执行：
- 代码修改后 → Stop hook 检查 transcript 中是否有 pytest 输出
- 无测试输出 → BLOCK，要求运行 `pytest tests/`
- 测试失败 → BLOCK，要求修复

### PostToolUse 自动验证（可选）

```json
{
  "matcher": "Edit",
  "hooks": [{
    "type": "command",
    "command": "python -c \"import py_compile; py_compile.compile('$CLAUDE_TOOL_INPUT_FILE_PATH', doraise=True)\"",
    "timeout": 10
  }]
}
```

每次编辑后自动检查 Python 语法，语法错误立即反馈。

## 组件 11: 子智能体编排（Subagent Orchestration）

### 当前使用模式

本项目已使用 Agent 工具进行并行探索和计划生成。

### 流程规范化

| 工作流步骤 | 子 agent 类型 | 隔离模式 |
|-----------|-------------|---------|
| Step 1 - Brainstorming | Skill 调用 | 无 |
| Step 2 - Propose | Skill 调用（openspec-propose） | 无 |
| Step 4 - Writing Plan | Agent (Plan) | 无 |
| Step 5 - Execute | Agent (general-purpose) | worktree |
| Step 9 - Debug | Agent (Explore → general-purpose) | 无 |

**关键原则**（来自文章）：子 agent 广泛探索，但只返回 1000-2000 token 压缩摘要。

## 组件 12: 七个设计决策

| # | 决策 | 本项目选择 | 依据 |
|---|------|----------|------|
| 1 | 单智能体 vs 多智能体 | **单智能体** | Anthropic/OpenAI 共识：先最大化单个 agent |
| 2 | ReAct vs 计划-执行 | **计划-执行** | 有 9 步工作流，天然适合计划-执行模式 |
| 3 | 上下文窗口管理 | **即时检索 + 结构化笔记** | 用 Grep/Glob 代替 cat，计划写入 docs/plans/ |
| 4 | 验证循环设计 | **计算验证为主** | pytest + py_compile，确定性真相 |
| 5 | 权限和安全 | **限制性** | PreToolUse 拦截，白名单放行 |
| 6 | 工具范围策略 | **默认工具集** | Claude Code 自带，不做额外限制 |
| 7 | Harness 厚度 | **薄** | 遵循 Anthropic 哲学：模型改进时删除 harness 复杂度 |

## 判断标准（来自文章）

> "如果性能随着更强大的模型扩展而不增加 harness 复杂性，设计就是合理的"

**Harness 不是目的，是手段。** 当新的 Claude 模型内化了当前需要 harness 强制执行的规则时，应主动删除对应的 hook 和脚本。
